# SUDOMG Design Principles

These principles guide every design decision within ActionArc.

---

## Dashboard over Wizard

Important information should be visible before it needs to be searched for.

Avoid unnecessary dialogs and multi-step wizards.

---

## Few Clicks

Common actions should never require excessive navigation.

The software should feel fast.

---

## Local First

ActionArc should function without cloud services whenever possible.

Cloud features should enhance—not replace—the local experience.

---

## Transparency

Users should understand:

- what is running
- why it ran
- when it ran
- what happened

Automation should never feel like magic.

---

## Build Once

Core functionality belongs in the Engine.

GUI, CLI, Services, and future APIs are simply different interfaces to the same engine.

---

## Grow With The User

Users should never need to switch products as they become more advanced.

Simple users should be productive immediately.

Advanced users should never feel constrained.

---

## Convenience is Premium

Premium should remove work.

Premium should never artificially remove capability.

Custom scripting should always remain a first-class citizen.

---

## Reusable Building Blocks

Everything should encourage reuse.

Triggers

Actions

Providers

Arcs

should all be reusable components.

---

## Clean Interfaces

Avoid clutter.

Prefer one good screen over many dialogs.

Hide complexity until it becomes necessary.

---

## Explainability

An Arc should be understandable by reading it.

Automation should be inspectable rather than mysterious.
